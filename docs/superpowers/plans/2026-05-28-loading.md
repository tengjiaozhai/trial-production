## 上传阶段 LLM 阻塞与流式进度改造计划

### Summary
- 目标：上传文件后立即进入全屏 `loading`，并在“上传阶段的所有 LLM 任务完成”前不允许用户点击“点此开始解析”。
- 根因：当前 LLM 调用在 `handleFileUpload` 异步执行，而“点此开始解析”只校验 Step1 必填，不校验上传解析是否完成，导致时序空值。
- 范围：按你确认的口径，等待上传阶段所有 LLM（核心器件匹配 + 关键物料匹配）完成后再放开。

### Implementation Changes
- 在 [App.tsx](/Users/shenmingjie/tinno/trial-production/trial-production/src/App.tsx) 增加上传解析专用状态：
  - `isUploadResolving`（上传解析是否进行中）
  - `loadingPhase`（`upload | calc`，区分上传和开始解析两个 loading 来源）
  - 复用 `loadingText` 作为流式步骤文本，新增 `loadingTitle` 或由 `loadingPhase` 映射标题文案。
- 重构 `handleFileUpload` 的执行骨架（`try/finally`）：
  - 入口立即 `setLoading(true)` + `setLoadingPhase('upload')` + 首条步骤。
  - 在关键节点推进流式步骤文本（真实阶段推进，不做固定延时）：
    - `读取上传文件`
    - `解析配置表并提取PCBA`
    - `解析管控物料表`
    - `核心器件大模型匹配中`
    - `关键物料大模型匹配中`（仅存在该文件时显示）
    - `写入解析结果`
  - 所有上传阶段异步任务结束后统一关闭 loading（`finally`），确保异常也能释放阻塞。
- 修改按钮与输入的放开条件：
  - “点此开始解析”按钮禁用条件改为：`!isStep1Complete || isUploadResolving`。
  - 上传区域 `<input type="file">` 在 `isUploadResolving` 时禁用，避免重复触发并发上传。
  - 若需要，删除文件/勾选 PCBA 的交互在 `isUploadResolving` 时同样禁用，保持状态一致性。
- 复用现有全屏 loading UI（[App.tsx](/Users/shenmingjie/tinno/trial-production/trial-production/src/App.tsx:1201)）：
  - `upload` 阶段标题示例：`文件上传解析中...`
  - `calc` 阶段保持原文案：`AI 正在深度解析...`
  - 文本区持续显示当前步骤，达到“像开始解析一样看到步骤变化”的体验。

### Public Interfaces / Types
- 不改外部 API。
- 仅新增页面内部状态与辅助函数，不变更 `types.ts` 的对外类型契约。

### Test Plan
- 手工场景 1：上传四个文件后立刻尝试点击“点此开始解析”，应被禁用，且看到上传流式步骤。
- 手工场景 2：上传完成后按钮自动放开，点击后 Step2 中核心器件字段应按已有数据链路填充。
- 手工场景 3：仅上传配置表/物料表（无关键物料模板）时，步骤应跳过“关键物料大模型匹配中”，不会卡住。
- 手工场景 4：网络慢/超时时，loading 保持到本轮上传任务结束（含超时返回），随后正常放开，不出现永久遮罩。
- 回归场景：Step1 的“开始解析”原有 loading 动画与步骤文案不被上传 loading 逻辑覆盖。

### Assumptions
- 默认口径已锁定：上传阶段等待所有 LLM 任务完成后才放开。
- 流式步骤基于真实执行进度推进，不引入额外固定等待时间（避免人为拉长上传耗时）。
